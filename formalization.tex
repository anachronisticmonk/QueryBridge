\section*{Formalization: jq $\to$ SQL Translation Correctness}

We formalize a translation from jq-style queries on JSON records to SQL queries on a columnar relational store, and prove that the translation preserves meaning. The full development is mechanized in Lean~4 (\texttt{Main.lean}); Lean names are given in \texttt{typewriter} where they match.

\subsection*{1. Data}

A user has three fields: a name, an age, and a role from $\text{Role} = \{\text{student}, \text{employee}, \text{retired}\}$. The same user is represented two ways:
\begin{itemize}
    \item \textbf{JSON} (\texttt{Juser}): record $\{\text{name} : \text{String},\ \text{age} : \mathbb{N},\ \text{role} : \text{Role}\}$. A JSON database is $\mathcal{JD} = \text{List}\,\mathcal{J}$ (\texttt{JDB}).
    \item \textbf{SQL} (\texttt{Suser}): tuple $\text{String} \times \mathbb{N} \times \text{Role}$. An SQL database $\mathcal{SD}$ (\texttt{SDB}) is three parallel column lists $(\text{names}, \text{ages}, \text{roles})$ — there is no constraint that the columns have equal length.
\end{itemize}

Conversions $\text{toS} : \mathcal{J} \to \mathcal{S}$ and $\text{toJ} : \mathcal{S} \to \mathcal{J}$ form an isomorphism: \texttt{toJ\_toS} and \texttt{toS\_toJ} establish $\text{toJ} \circ \text{toS} = \text{id}$ and $\text{toS} \circ \text{toJ} = \text{id}$.

The columnar layout is unfolded into a row list by a 3-way zip (\texttt{SDB.toRows}):
\[
\text{toRows}(SD) = \text{zip}_3(SD.\text{names},\, SD.\text{ages},\, SD.\text{roles}).
\]
The zip stops at the shortest column, so only triples with all three components contribute rows. All semantics on $\mathcal{SD}$ go through $\text{toRows}$.

\subsection*{2. Schema, Values, and Conditions}

A column tag $\text{Col} = \{\text{name}, \text{age}, \text{role}, \text{all}\}$ identifies an attribute (with $\text{all}$ as a wildcard for select-everything queries). A value
\[
\text{Value} \;=\; \mathbb{N} \;\dot\cup\; \text{String} \;\dot\cup\; \text{Role}
\]
is a tagged union; comparison operators are $\text{Op} = \{=, >, <, \geq, \leq\}$.

A condition $C$ is generated by the grammar
\[
C \;::=\; \text{always} \;\mid\; \text{cmp}(\text{col}, \text{op}, v) \;\mid\; C \land C \;\mid\; C \lor C
\]
and interpreted on both record types as $\text{eval}_J(C, j)$ (\texttt{Cond.evalJ}) and $\text{eval}_S(C, s)$ (\texttt{Cond.evalS}). A type-mismatched comparison (e.g., $>$ between a $\text{Role}$ and a $\mathbb{N}$) returns $\text{false}$.

\begin{lemma}[Condition bridge] \label{lem:cond-bridge}
For every $C$, $\text{eval}_J(C, j) = \text{eval}_S(C,\, \text{toS}(j))$ \emph{(\texttt{eval\_bridgeJ})}, and dually $\text{eval}_S(C, s) = \text{eval}_J(C,\, \text{toJ}(s))$ \emph{(\texttt{eval\_bridgeS})}.
\end{lemma}
Proved by induction on $C$, using that column projection commutes with $\text{toS}$ (\texttt{getVal\_bridge}).

\paragraph{Per-row update.} The unconditional updater $\text{applyUpdate}_J(\text{col}, v, j)$ writes $v$ into the column \emph{only when the value's tag matches the column's payload type}; otherwise the row passes through unchanged. The conditional version $\text{applyUpdateIf}_J(\text{col}, v, C, j)$ updates iff $\text{eval}_J(C, j) = \text{true}$, with an analogous pair on $\mathcal{S}$.

\begin{lemma}[Update bridge] \label{lem:update-bridge}
$\text{applyUpdateIf}_S(\text{col}, v, C, \text{toS}(j)) = \text{toS}(\text{applyUpdateIf}_J(\text{col}, v, C, j))$ \emph{(\texttt{applyUpdateIf\_bridge})}.
\end{lemma}

\subsection*{3. Equivalence}

We use two relations between $JD \in \mathcal{JD}$ and $SD \in \mathcal{SD}$:
\begin{itemize}
    \item \emph{Set equivalence} $JD \equiv SD$ (\texttt{equiv}): same elements modulo $\text{toS}$/$\text{toJ}$, ignoring multiplicities.
    \item \emph{Permutation equivalence} $JD \approx SD$ (\texttt{permEquiv}): $JD\,\text{.map}\,\text{toS}$ is a permutation of $\text{toRows}(SD)$.
\end{itemize}
Permutation equivalence is strictly stronger.

\begin{lemma}[Permutation implies set] \label{lem:perm-set}
$JD \approx SD \implies JD \equiv SD$ \emph{(\texttt{permEquiv\_implies\_equiv})}.
\end{lemma}

\subsection*{4. Query Languages and Translation}

Each language has six constructors. The translation $T$ (\texttt{jquery\_to\_squery}) is structure-preserving; note the asymmetric naming for the aggregate (jq's operator is \texttt{length}, SQL's is \texttt{count}, but they compute the same thing).

\begin{center}
\begin{tabular}{lll}
\hline
jq (\texttt{JQuery}) & SQL (\texttt{SQuery}) & Result\\
\hline
$\texttt{find}(\text{col}, C)$       & $\texttt{select}(\text{col}, C)$       & database \\
$\texttt{drop}(C)$                   & $\texttt{delete}(C)$                   & database \\
$\texttt{prepend}(j)$                & $\texttt{insert}(\text{toS}(j))$       & database \\
$\texttt{clear}$                     & $\texttt{truncate}$                    & database \\
$\texttt{length}$                    & $\texttt{count}$                       & $\mathbb{N}$ \\
$\texttt{modify}(\text{col}, v, C)$  & $\texttt{update}(\text{col}, v, C)$    & database \\
\hline
\end{tabular}
\end{center}

\paragraph{Semantics.}
The two evaluators \texttt{eval\_jquery} and \texttt{eval\_squery} implement these clauses as expected: \texttt{find}/\texttt{select} filter; \texttt{drop}/\texttt{delete} filter the negation; \texttt{prepend}/\texttt{insert} cons at the head; \texttt{clear}/\texttt{truncate} return empty; \texttt{length}/\texttt{count} return list length; \texttt{modify}/\texttt{update} map a per-row updater. On the SQL side, every database-returning query decomposes columns into a row list via $\text{toRows}$, transforms the rows, then rebuilds three columns.

\paragraph{Result equivalence.}
The result equivalence \texttt{result\_equiv} dispatches by tag: two database results compare under $\equiv$ (set equivalence), two scalar results compare with $=$, and a tag mismatch is $\bot$. Note that the conclusion uses set equivalence even though the precondition is permutation equivalence — the proof uses permutation internally but only asserts set equivalence externally for non-aggregate cases.

\subsection*{5. Main Theorem}

\begin{theorem}[\texttt{query\_equiv}] \label{thm:main}
If $JD \approx SD$, then for any jq query $q$,
\[
\text{eval}_{jq}(JD, q) \;\equiv_R\; \text{eval}_{sql}(SD, T(q)).
\]
\end{theorem}

\begin{proof}[Proof sketch]
Case analysis on $q$, with three reusable ingredients:
\begin{itemize}
    \item Lemma~\ref{lem:perm-set} downgrades the hypothesis to set equivalence for non-aggregate cases.
    \item Lemma~\ref{lem:cond-bridge} keeps filter and update predicates in lockstep across formats.
    \item Round-trip lemmas show that the SQL-side decompose-transform-rebuild is invisible to $\text{toRows}$: \texttt{toRows\_filter\_reconstruct} for filters, \texttt{toRows\_map\_reconstruct} for maps, \texttt{toRows\_insert} for cons. All reduce to \texttt{zip3\_map\_components}, the identity $\text{zip}_3(\ell\,\text{.map}\,\pi_1, \ell\,\text{.map}\,\pi_{2,1}, \ell\,\text{.map}\,\pi_{2,2}) = \ell$.
\end{itemize}
The six cases:
\begin{itemize}
    \item \emph{\texttt{find}, \texttt{drop}}: membership reasoning, using Lemmas~\ref{lem:perm-set}, \ref{lem:cond-bridge}, and the filter round-trip.
    \item \emph{\texttt{prepend}}: case-split on whether a record is the prepended one or already present, using the insert round-trip and $\text{toS}$/$\text{toJ}$ inversion.
    \item \emph{\texttt{clear}}: both sides reduce to empty membership.
    \item \emph{\texttt{length}}: this case genuinely needs $\approx$. Length preservation under permutation gives $|JD\,\text{.map}\,\text{toS}| = |\text{toRows}(SD)|$, and $\text{.map}$ preserves length.
    \item \emph{\texttt{modify}}: chains the update bridge (Lemma~\ref{lem:update-bridge}) applied pointwise via $\text{map} \circ \text{map}$, then permutation transport under $\text{.map}$, then the map round-trip; finally Lemma~\ref{lem:perm-set} lifts the resulting $\approx$ to the asserted $\equiv$.
\end{itemize}
Only \texttt{length} and \texttt{modify} genuinely consume the strength of $\approx$; the rest factor through $\equiv$ alone.
\end{proof}

\subsection*{6. Mechanization}

All definitions, lemmas, and the main theorem are mechanized in \texttt{Main.lean}. Property-based tests in \texttt{Tests.lean} use Plausible to stress-test the same correspondence on randomly generated databases and queries. \texttt{Error.lean} contains a deliberately-bugged variant; running the tests against it surfaces minimal counterexamples, demonstrating exactly the class of bug the proof rules out.
