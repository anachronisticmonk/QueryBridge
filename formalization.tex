\section*{Formal Specification: JQ $\to$ SQL Correctness}

This document formalizes a translation from jq-style queries over JSON data into SQL-style queries over a \emph{columnar} relational store, and proves that the translation preserves meaning.

% =====================================================

\subsection*{1. Data Structures and Domains}

This section defines the two representations of the same logical data: a user database stored as JSON records and as a columnar SQL store.

Let $\mathcal{J}$ be the domain of JSON users, where each $j \in \mathcal{J}$ is a record:
\[
j = \{ \text{name} : \text{String}, \ \text{age} : \mathbb{N} \}.
\]

Let $\mathcal{S}$ be the domain of SQL users, where each $s \in \mathcal{S}$ is a tuple in the product type:
\[
\mathcal{S} \;=\; \text{String} \times \mathbb{N}.
\]

\begin{itemize}
    \item \textbf{JSON Database ($\mathcal{JD}$):} A finite list of JSON user records,
    \[
        \mathcal{JD} \;=\; \text{List}\,\mathcal{J}.
    \]
    \item \textbf{SQL Database ($\mathcal{SD}$):} A \emph{columnar} record consisting of two parallel lists, one per attribute:
    \[
        SD \;=\; \big(\,\text{names} : \text{List String},\ \text{ages} : \text{List }\mathbb{N}\,\big).
    \]
    The two columns are aligned positionally: the $i$-th logical row of $SD$ is the pair $(SD.\text{names}[i],\,SD.\text{ages}[i])$. The columns are not assumed to have equal length; only the common prefix gives rise to logical rows.
\end{itemize}

\paragraph{Row projection.}
To bridge the columnar storage with a relational view we define
\[
\text{toRows} : \mathcal{SD} \to \text{List}\,\mathcal{S},
\qquad
\text{toRows}(SD) \;=\; \text{zip}\big(SD.\text{names},\ SD.\text{ages}\big),
\]
which returns the list of $(\text{name}, \text{age})$ tuples encoded by the two columns. All semantic statements about $SD$ are mediated through $\text{toRows}(SD)$.

\paragraph{Conversion functions.}
Between $\mathcal{J}$ and $\mathcal{S}$ we have a structural isomorphism:
\[
\text{toS} : \mathcal{J} \to \mathcal{S}, \qquad
\text{toJ} : \mathcal{S} \to \mathcal{J},
\]
with
\[
\text{toS}\big(\{\text{name}=n,\ \text{age}=a\}\big) \;=\; (n, a),
\qquad
\text{toJ}\big((n,a)\big) \;=\; \{\text{name}=n,\ \text{age}=a\}.
\]
These satisfy $\text{toJ} \circ \text{toS} = \text{id}_{\mathcal{J}}$ and $\text{toS} \circ \text{toJ} = \text{id}_{\mathcal{S}}$.

% =====================================================

\subsection*{2. Database Equivalence}

This section defines when a JSON database and a columnar SQL database represent the same information.

Two databases $JD \in \mathcal{JD}$ and $SD \in \mathcal{SD}$ are \emph{equivalent}, written $JD \equiv SD$, when their contents agree up to conversion. Because $SD$ is columnar, membership on the SQL side is taken with respect to its row projection $\text{toRows}(SD)$:

\begin{align*}
JD \equiv SD \iff{} & \big( \forall j \in \mathcal{J},\ j \in JD \leftrightarrow \text{toS}(j) \in \text{toRows}(SD) \big) \\
                  \land\ & \big( \forall s \in \mathcal{S},\ s \in \text{toRows}(SD) \leftrightarrow \text{toJ}(s) \in JD \big).
\end{align*}

Intuitively, every JSON record exists in the database if and only if its tuple encoding occurs as a row of the columnar SQL database, and vice versa.

% =====================================================

\subsection*{3. Condition Semantics}

This section defines how conditions (e.g.\ \texttt{age > 20}) are evaluated in both representations.

A condition $C$ is built from constants, atomic comparisons over a column, and Boolean combinations $\land,\lor$. It is interpreted over both formats:

\begin{itemize}
    \item $\text{eval}_J(C, j)$: evaluates $C$ on a JSON record $j$ via field access on the record;
    \item $\text{eval}_S(C, s)$: evaluates $C$ on an SQL tuple $s$ via projection of the tuple components.
\end{itemize}

\paragraph{Bridge Lemma.}
This lemma ensures condition evaluation is consistent across representations.

\begin{lemma}[Condition Bridge]
For every condition $C$ and every $j \in \mathcal{J}$,
\begin{equation}
\text{eval}_J(C, j) \;=\; \text{eval}_S\big(C,\ \text{toS}(j)\big).
\end{equation}
Equivalently, for every $s \in \mathcal{S}$, $\text{eval}_S(C, s) = \text{eval}_J(C, \text{toJ}(s))$.
\end{lemma}

This is the key semantic consistency property between the JSON and SQL worlds, and it is established by induction on $C$ using the corresponding bridge for column projection.

% =====================================================

\subsection*{4. Translation Function}

This section defines the mapping from jq queries to SQL queries.

\[
T : Q_{jq} \to Q_{sql}
\]

The translation preserves the structure of the query while changing its target representation:

\begin{itemize}
    \item \texttt{find} $\rightarrow$ \texttt{SELECT}
    \item \texttt{drop} $\rightarrow$ \texttt{DELETE}
\end{itemize}

The condition $C$ is carried through unchanged; only the interpretation of $C$ shifts from $\text{eval}_J$ to $\text{eval}_S$.

% =====================================================

\subsection*{5. Query Semantics}

This section defines how queries operate as filters over databases. Both \texttt{find}/\texttt{SELECT} (keep matches) and \texttt{drop}/\texttt{DELETE} (keep non-matches) are filters; only the predicate differs.

\paragraph{jq semantics.}
A jq query filters the JSON list directly:
\[
\text{eval}_{jq}(JD,\ \texttt{find}\,C) = \{\, j \in JD \mid \text{eval}_J(C, j) = \text{true} \,\},
\]
\[
\text{eval}_{jq}(JD,\ \texttt{drop}\,C) = \{\, j \in JD \mid \text{eval}_J(C, j) = \text{false} \,\}.
\]

\paragraph{Columnar SQL semantics.}
Because $SD$ is stored as two parallel columns, evaluation proceeds in three steps: \emph{(i)} project the columnar store to its row view, \emph{(ii)} filter the rows by the condition, and \emph{(iii)} reconstruct the two columns from the filtered rows. Writing $r = \text{toRows}(SD)$ and letting
\[
r_C = \{\, s \in r \mid \text{eval}_S(C, s) = \text{true} \,\},
\quad
r_{\lnot C} = \{\, s \in r \mid \text{eval}_S(C, s) = \text{false} \,\},
\]
we define
\[
\text{eval}_{sql}(SD,\,\texttt{SELECT}\,C)
= \big(\text{names} = \pi_1^{*}(r_C),\ \text{ages} = \pi_2^{*}(r_C)\big),
\]
\[
\text{eval}_{sql}(SD,\,\texttt{DELETE}\,C)
= \big(\text{names} = \pi_1^{*}(r_{\lnot C}),\ \text{ages} = \pi_2^{*}(r_{\lnot C})\big),
\]
where $\pi_1^{*}$ and $\pi_2^{*}$ are the pointwise first- and second-component projections (i.e.\ $\pi_1^{*}(\ell) = \ell.\text{map}(\pi_1)$). The result is again a columnar SQL database in $\mathcal{SD}$.

\paragraph{Columnar Round-Trip Lemma.}
The decomposition--filter--reconstruction discipline is sound:

\begin{lemma}[Round-trip]
\label{lem:roundtrip}
For every $SD \in \mathcal{SD}$ and every predicate $p : \mathcal{S} \to \text{Bool}$,
\[
\text{toRows}\!\Big(\big(\text{names} = \pi_1^{*}(r_p),\ \text{ages} = \pi_2^{*}(r_p)\big)\Big)
\;=\;
r_p,
\quad\text{where } r_p = \{\,s \in \text{toRows}(SD) \mid p(s)\,\}.
\]
\end{lemma}

\noindent The proof reduces to $\text{zip}(\ell.\text{map}(\pi_1),\ \ell.\text{map}(\pi_2)) = \ell$ for any list of pairs $\ell$, by induction on $\ell$. As a consequence, both jq and SQL queries behave as filters over the same logical data, just observed through different storage layouts.

% =====================================================

\subsection*{6. Main Correctness Theorem}

This theorem states that translation preserves meaning across both database systems.

\begin{theorem}[Translation Correctness]
Let $JD \in \mathcal{JD}$ and $SD \in \mathcal{SD}$ with $JD \equiv SD$.
For any jq query $q$, executing $q$ on $JD$ and the translated query $T(q)$ on $SD$ yields equivalent databases:
\[
\text{eval}_{jq}(JD,\,q)
\;\equiv\;
\text{eval}_{sql}\big(SD,\,T(q)\big).
\]
\end{theorem}

\begin{proof}
We prove both directions of the equivalence, separately for the \texttt{find} and \texttt{drop} cases. Let $C$ be the condition of $q$ and write $r = \text{toRows}(SD)$. Throughout, the Round-trip Lemma~\ref{lem:roundtrip} identifies the row projection of the SQL output with the corresponding filtered row list ($r_C$ or $r_{\lnot C}$).

\smallskip\noindent\emph{Find case: $q = \texttt{find}\,C$, $T(q) = \texttt{SELECT}\,C$.}
For any $j \in \mathcal{J}$,
\[
j \in \text{eval}_{jq}(JD,\,\texttt{find}\,C)
\iff
j \in JD \,\land\, \text{eval}_J(C, j) = \text{true}.
\]
By $JD \equiv SD$ we have $j \in JD \iff \text{toS}(j) \in r$, and by the Condition Bridge Lemma, $\text{eval}_J(C, j) = \text{eval}_S(C, \text{toS}(j))$. Combining,
\[
j \in \text{eval}_{jq}(JD,\,\texttt{find}\,C)
\iff
\text{toS}(j) \in r_C.
\]
By the Round-trip Lemma, $r_C = \text{toRows}\!\big(\text{eval}_{sql}(SD,\,\texttt{SELECT}\,C)\big)$, so
\[
j \in \text{eval}_{jq}(JD,\,\texttt{find}\,C)
\iff
\text{toS}(j) \in \text{toRows}\!\big(\text{eval}_{sql}(SD,\,\texttt{SELECT}\,C)\big).
\]
The symmetric direction  that every row of the SQL result corresponds to a JSON record in the jq result  follows by applying the same argument to $\text{toJ}$, using the second clause of $\equiv$ and the dual form of the Bridge Lemma.

\smallskip\noindent\emph{Drop case: $q = \texttt{drop}\,C$, $T(q) = \texttt{DELETE}\,C$.}
The argument is identical, with every occurrence of $\text{eval}(C, \cdot) = \text{true}$ replaced by $\text{eval}(C, \cdot) = \text{false}$, $r_C$ replaced by $r_{\lnot C}$, and \texttt{SELECT} by \texttt{DELETE}. The Bridge Lemma still equates the JSON and SQL predicates pointwise, and the Round-trip Lemma still identifies the SQL output with $r_{\lnot C}$ under $\text{toRows}$.

\smallskip
Combining both directions in both cases yields $\text{eval}_{jq}(JD,\,q) \equiv \text{eval}_{sql}(SD,\,T(q))$.
\end{proof}

% =====================================================

\subsection*{7. Interpretation}

The theorem guarantees semantic preservation across a non-trivial change of representation: the jq side stores data row-by-row as records, while the SQL side stores it column-by-column as parallel lists. The translation $T$ is semantics-preserving in the sense that, despite this representational shift, every jq query has a SQL counterpart that produces an equivalent result.

Two ingredients carry the proof:
\begin{itemize}
    \item the \emph{Condition Bridge Lemma}, which keeps predicate evaluation in lockstep across $\mathcal{J}$ and $\mathcal{S}$;
    \item the \emph{Columnar Round-trip Lemma}, which keeps the columnar storage discipline transparent to all higher-level reasoning by reducing it to filtering on rows.
\end{itemize}

Together they show that columnar storage is invisible at the level of query semantics: any property provable for the row-based view of $SD$ lifts to $SD$ itself.
